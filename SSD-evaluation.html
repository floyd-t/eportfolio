Introduction

This document provides supporting evidence and analysis referenced within the accompanying reflection, including system outputs, testing results and implementation details.
This e-portfolio documents the design, development and evaluation of a secure online retailer system, developed as part of the Secure Software Development module. The project aims to demonstrate the application of secure coding principles through the implementation of a command-line based application in Python, with the ability to spontaneously enable and disable security controls.
The system was initially designed collaboratively, with a focus on integrating security mechanisms such as authentication, input validation and attack mitigation into the core architecture. The final implementation was completed individually, translating the design into a functional system using object-oriented programming, the Model–View–Controller (MVC) pattern and the Strategy Pattern to modularise security behaviour.
This portfolio presents the artefacts produced throughout development, including system design, implementation and testing evidence. It also evaluates the extent to which the final system meets the original design specification and reflects on the development process, highlighting key learning outcomes and challenges encountered.

Artefacts and Development

The development of the secure online retailer system was supported by a range of artefacts produced throughout the module. The initial design phase (Unit 6) included UML diagrams such as a class diagram, sequence diagram and an activity diagram. These artefacts provided a structured representation of system components, user interactions and potential security threats, forming the foundation for implementation.
The final system (Unit 11) was developed in Python using the PyCharm integrated development environment. The codebase follows the Model–View–Controller (MVC) architecture, ensuring separation of concerns between data models, business logic and user interaction. Core components include controllers for authentication, product management and order processing, alongside models representing system entities such as users and orders.
A key feature of the implementation is the use of the Strategy Pattern to encapsulate security behaviour. The system defines both secure and insecure strategies, allowing security mechanisms such as input validation, password hashing and rate limiting to be toggled dynamically at runtime. This modular approach improves maintainability and enables direct comparison between secure and vulnerable system states.
The project is structured into logical modules, including controllers, models, security, attacks and data, ensuring clarity and scalability. Inline code comments and adherence to PEP-8 standards further enhance code readability and maintainability.

Security testing formed a key part of the validation process. Simulated attacks were implemented to assess system resilience under both secure and insecure configurations. A brute force attack simulation demonstrated the effectiveness of rate limiting and account lockout mechanisms in secure mode, while highlighting system vulnerability when these controls were disabled. Similarly, an injection attack simulation tested input validation by attempting to introduce malicious input into the system. In secure mode, validation successfully rejected the input, whereas in insecure mode the attack was accepted, demonstrating the importance of input verification. Requirements and Assumptions

The system was developed based on a set of functional and security requirements derived from the assignment brief, alongside assumptions made during the design process. Core functional requirements included user authentication, product management and order processing, all of which were implemented through a command-line interface. These features were designed to reflect a simplified online retailer system while maintaining a focus on secure interaction.
Security requirements played a central role in shaping the system design. The need to mitigate common threats such as brute force attacks, injection attacks and denial-of-service scenarios (OWASP, n.d.) informed the inclusion of mechanisms such as input validation, rate limiting, password hashing and event logging. These decisions align with widely recognised security frameworks such as OWASP, which emphasise the importance of reducing attack surfaces and validating all external input (OWASP, 2025).
Several assumptions were made to support development. It was assumed that the system would operate in a controlled environment, allowing attacks to be simulated rather than executed at a network level. This influenced the design of denial-of-service mitigation, which was implemented through request throttling rather than full network simulation. Additionally, the use of a simulated external API for payment processing reflects an assumption that sensitive financial data should not be handled directly by the system, thereby reducing security risk.
Overall, the requirements and assumptions guided the development of a system that balances functional simplicity with meaningful security implementation, enabling clear demonstration of secure and insecure behaviours.

Evaluation of Final Implementation Against Design Proposal

The final implementation of the secure online retailer system largely aligns with the original design proposal developed in Unit 6. The system successfully completes the core architectural, functional and security requirements outlined during the design phase, with several refinements made during development to improve robustness and ease of use.

Architectural Alignment

The system adheres closely to the Model–View–Controller (MVC) architectural pattern defined in the design proposal. The MVC pattern was chosen mainly due to the level of ease it provides in extending or changing code during development (Gamma, E. et al., 1994). The Model layer is implemented through domain classes such as Customer, Product and Order, which encapsulate system state and enforce controlled data access. The View is represented by a command-line interface, enabling user interaction with the system, while the Controller layer manages business logic, including authentication and order processing.
In addition to MVC, the Strategy Pattern was implemented as planned to modularise security behaviour. The introduction of SecureStrategy and InsecureStrategy allows the system to dynamically switch between secure and insecure configurations during runtime. This design decision proved highly effective, enabling direct comparison between protected and vulnerable system states, thereby fulfilling the requirement to demonstrate the impact of security controls. This flexibility during runtime was the main reason it was chosen, the major disadvantage of Strategy Pattern (how it struggles with larger amounts of classes)(Stackify, 2025) would not be a problem with this system due to its limited size.

Security Features and Enhancements

The implementation successfully incorporates the required security mechanisms, including authentication, authorisation, input validation, logging and session management. Password hashing using SHA-256 is applied in secure mode, while insecure mode intentionally stores passwords in plain text to demonstrate vulnerability.
During development, additional refinements were introduced beyond the original design. Notably, account lockout functionality was added to complement rate limiting, ensuring that brute force attacks are mitigated even if correct credentials are eventually attempted. This enhancement strengthens the system’s defensive capabilities and reflects real-world security practices.

Handling of Security Modes

A key design requirement was the ability to toggle security features on and off. This was successfully implemented through a centralised configuration component that dynamically assigns the appropriate security strategy. During development, an inconsistency was identified where shared data storage between secure and insecure modes caused mismatches in password verification. This issue was resolved by isolating system state per security configuration, ensuring consistent behaviour across modes.

Functional Requirements

All core functional requirements were met. The system supports user registration, authentication and order placement, fulfilling the required CRUD capabilities. The use of Python data structures aligns with the design specification, with dictionaries used for user storage, lists for order tracking and sets for session management.

Limitations and Deviations

While the system meets the majority of design requirements, certain limitations remain. The denial-of-service attack is simulated rather than implemented at a network level, due to the constraints of a command-line environment. Additionally, while encryption is represented through hashing, full end-to-end encryption mechanisms were not implemented.

Summary

Overall, the final implementation closely reflects the original design while incorporating meaningful improvements identified during development. The system not only meets the specified requirements but also demonstrates a deeper understanding of secure system design through the application of layered security controls and modular architecture.
