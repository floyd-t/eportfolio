Secure Retail System e-Portfolio Reflection

This reflection should be read alongside the accompanying evaluation document, which contains detailed evidence of system implementation, testing outputs and security demonstrations referenced throughout this reflection.

The development of this secure online retail system provided an opportunity to apply both theoretical and practical aspects of secure software development. This project required not only the implementation of functional features, but also the integration of security principles into the design and development process. Reflecting on this experience, several key areas of learning emerged, including problem-solving and the challenges of maintaining consistency across different system configurations.

The project began as a collaborative effort during the design phase, where the team collectively contributed to the development of the system architecture. Rather than assigning fixed roles, responsibilities were shared and developed gradually, which allowed for a broader understanding of the system as a whole. This approach supported collaborative learning, although it also highlighted the importance of clearer task allocation in future projects to improve efficiency. Transitioning from the group-based design phase to individual implementation required careful interpretation of shared design artefacts, reinforcing the need for clear and well-documented system designs.

One of the most significant learning experiences occurred during the implementation of the security strategy. The use of the Strategy Pattern to toggle between secure and insecure modes initially appeared straightforward, however practical implementation revealed complexities that were not fully anticipated during the design phase. A critical issue arose when both secure and insecure modes shared the same data repository, leading to inconsistencies in password verification. Specifically, secure mode stored hashed passwords, while insecure mode used plain text, resulting in failed authentication when switching modes. Identifying and resolving this issue required a deeper understanding of how the internal system state interacts with security mechanisms. The solution involved isolating data repositories for each mode, ensuring consistency in how data was stored and validated. This experience emphasised the importance of aligning data handling strategies with security configurations, as well as highlighting how seemingly small design decisions can have significant implications for system behaviour.

Another key challenge involved implementing effective protection against brute force attacks. While initial designs included rate limiting, testing revealed that this alone was insufficient, as successful authentication could still occur if the correct password was attempted within the allowed threshold. To address this, account lockout functionality was introduced, preventing further login attempts after a defined number of failures. This improvement demonstrated the importance of iterative development and testing in identifying weaknesses in security mechanisms. It also reinforced the concept of defence-in-depth, where multiple layers of protection are required to effectively mitigate attacks (Fortinet, 2025).

Time management was an additional challenge throughout the development process. Balancing the implementation of core functionality with the integration of security features required careful prioritisation. In particular, debugging and testing stages took longer than anticipated, especially when addressing issues such as inconsistent behaviour between security modes. This highlighted the importance of allocating sufficient time for testing and refinement, rather than focusing solely on initial implementation. In future projects, adopting a more structured development schedule with dedicated testing phases would improve overall efficiency and reduce last-minute problem solving.

Testing itself proved to be both a challenge and a valuable learning experience. While functional testing was relatively straightforward, designing effective security tests required a more analytical approach. Simulating attacks such as brute force and injection attacks required consideration of how an attacker might interact with the system, rather than simply verifying expected behaviour. This shift in perspective was particularly important in understanding how vulnerabilities can arise and how they can be mitigated. The ability to toggle security features on and off provided a clear demonstration of the effectiveness of implemented controls, reinforcing the importance of comparative testing in secure software development.

From a technical perspective, the project strengthened my understanding of object-oriented programming and design patterns. Implementing the Model–View–Controller architecture helped to separate concerns within the system, improving code organisation and maintainability. The use of the Strategy Pattern further enhanced modularity, allowing security behaviour to be encapsulated and modified without affecting the rest of the system. These design decisions not only improved the structure of the application, but also made it easier to test and extend.

In terms of security knowledge, the project deepened my understanding of common vulnerabilities and mitigation techniques, particularly those outlined in frameworks such as OWASP (OWASP, n.d.). Concepts such as input validation, rate limiting, and secure password handling were not only implemented, but also evaluated in practice. This practical application helped to bridge the gap between theoretical knowledge and real-world implementation, highlighting the challenges of designing systems that are both functional and secure.

Overall, the development process was completed in segments with repeated tests throughout to ensure each piece was functioning as intended, multiple refinements were made in response to testing and identified issues. This iterative approach proved essential in improving both functionality and security, it demonstrated the value of continuous evaluation throughout the software development lifecycle. This experience has reinforced the importance of integrating security considerations from the earliest stages of design, rather than treating them as something to design later during the development lifecycle.

Screenshots of the code and the results from testing have been placed in the Evaluation document submitted alongside this reflection.

In conclusion, this project has significantly enhanced my understanding of secure software development, both in terms of technical implementation and critical thinking. The challenges encountered during development provided valuable insights into system design, security and testing. The challenges have also highlighted areas for improvement in future work. The combination of collaborative design and individual implementation allowed for both shared learning and personal development, resulting in a well-rounded and practical learning experience.
